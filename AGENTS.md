# AGENTS.md

Постоянные инструкции для AI-агентов, работающих над проектом `ai-gram`.

Этот файл является базовым контрактом между владельцем проекта и Codex. Он не заменяет конкретное задание в текущем промпте. Если текущий промпт содержит более точные требования, выполняй его, но не нарушай правила из этого файла.

## 1. Роль агента

Ты — senior Go-разработчик и архитектор библиотек.

Ты работаешь над библиотекой для Telegram Bot API на Go. Цель — создать аккуратную, расширяемую, тестируемую и удобную библиотеку, а не просто набор функций поверх HTTP.

Работай как инженер, который оставляет после себя поддерживаемый код:

- простая архитектура важнее преждевременной универсальности;
- публичный API должен быть удобным и стабильным;
- внутренние детали должны быть отделены от пользовательского интерфейса библиотеки;
- каждый этап должен завершаться рабочим, проверяемым состоянием проекта.

## 2. Главная цель проекта

Проект `ai-gram` — это Go-библиотека для работы с Telegram Bot API.

Библиотека должна позволять разработчику:

- создавать клиента бота;
- вызывать методы Telegram Bot API типобезопасным способом;
- получать обновления через long polling;
- принимать webhook-обновления;
- строить обработчики команд, сообщений, callback query и других update-событий;
- подключать middleware;
- тестировать ботов без реального Telegram;
- расширять библиотеку при появлении новых методов Bot API.

Проект не должен превращаться во фреймворк с лишней магией. Сначала библиотека, потом удобные надстройки.

## 5. Архитектурные принципы

### 5.1. Разделяй уровни

Не смешивай в одном месте:

- типы Telegram Bot API;
- HTTP-клиент;
- transport long polling;
- webhook server/handler;
- router/dispatcher;
- middleware;
- retry/rate limit;
- observability;
- test utilities.

Каждый уровень должен иметь ясную ответственность.

### 5.2. API-клиент не должен знать про dispatcher

Пакет, отвечающий за вызовы Telegram Bot API, не должен зависеть от роутера, middleware, long polling или webhook.

Клиент должен уметь:

- принимать `context.Context`;
- отправлять запросы к Telegram;
- декодировать ответы;
- возвращать типизированные ошибки;
- работать через настраиваемый `http.Client` или интерфейс транспорта.

### 5.3. Dispatcher не должен знать детали HTTP API

Dispatcher работает с уже полученными `Update` и вызывает обработчики.

Он не должен самостоятельно ходить в Telegram API, кроме случаев, когда это явно предусмотрено отдельным слоем или переданным интерфейсом.

### 5.4. Transport должен быть заменяемым

Long polling и webhook — это способы доставки updates.

Они должны быть реализованы так, чтобы пользователь мог выбрать нужный способ, а тесты могли подменить источник updates.

### 5.5. Типы Bot API — фундамент библиотеки

Типы Telegram Bot API должны быть аккуратными, предсказуемыми и максимально совместимыми с JSON Telegram.

Не добавляй поведение в типы без необходимости. Типы в первую очередь описывают контракт данных.

## 6. Предпочтительная структура пакетов

Не создавай все пакеты сразу без необходимости. Структура ниже — ориентир, а не требование создать пустые директории.

Возможная структура:

```text
.
├── aiogram.go              # основной фасадный пакет, если нужен
├── api/                    # Bot API client, request/response handling
├── types/                  # Telegram Bot API types
├── transport/
│   ├── longpoll/           # long polling update source
│   └── webhook/            # webhook update receiver
├── dispatch/               # router, handlers, update dispatching
├── middleware/             # middleware contracts and common middleware
├── files/                  # upload/download helpers
├── observability/          # logging/tracing hooks without heavy dependency
├── testkit/                # test helpers, fake API, fake update source
├── examples/               # small runnable examples
└── internal/               # implementation details not for public API
```

Правила структуры:

- не клади всё в один большой пакет;
- не создавай циклические зависимости;
- `types` не должен импортировать высокоуровневые пакеты проекта;
- `api` может зависеть от `types`;
- `transport` может зависеть от `api` и `types`, если это оправдано;
- `dispatch` должен зависеть от `types`, но не обязан зависеть от `api`;
- `testkit` может зависеть от публичных пакетов.

## 7. Публичный API

Публичный API должен быть:

- маленьким;
- понятным;
- идиоматичным для Go;
- стабильным;
- удобным для автодополнения в IDE.

Не экспортируй сущности “на всякий случай”.

Экспортируемые Go-символы должны иметь комментарии в стиле GoDoc.

Плохой подход:

```go
type Bot struct {
    Token string
    Client *http.Client
    Debug bool
    InternalCache map[string]any
}
```

Лучше:

```go
type Client struct {
    // private fields
}

func NewClient(token string, opts ...Option) *Client
```

Внешние пользователи не должны зависеть от внутренних деталей реализации.

## 8. Ошибки

Ошибки должны быть полезными для пользователя библиотеки.

Требования:

- не теряй исходную ошибку;
- используй wrapping через `%w`;
- для ошибок Telegram API сделай отдельный тип ошибки;
- пользователь должен иметь возможность определить, что Telegram вернул `ok: false`;
- не паникуй в библиотечном коде, кроме случаев грубого нарушения внутреннего инварианта.

Желательная модель:

```go
type APIError struct {
    Code        int
    Description string
    Parameters  *ResponseParameters
}

func (e *APIError) Error() string
```

## 9. Context и отмена операций

Все операции, которые могут блокироваться, обязаны принимать `context.Context`.

Это касается:

- HTTP-запросов;
- long polling;
- webhook shutdown;
- dispatcher loop;
- file download/upload;
- retry/backoff ожиданий.

Не используй `context.Background()` внутри библиотечных операций, если context должен прийти от пользователя.

## 10. Конкурентность

Код должен быть безопасен и предсказуем.

Правила:

- явно документируй, какие типы безопасны для конкурентного использования;
- не запускай goroutine без понятного жизненного цикла;
- каждая goroutine должна завершаться по context cancellation или shutdown;
- не допускай утечек goroutine;
- не блокируй dispatcher навсегда из-за одного handler;
- panics в пользовательских handlers должны быть перехвачены recovery middleware или явно задокументированы.

## 11. Telegram Bot API

При реализации методов и типов Telegram Bot API ориентируйся на официальный контракт Telegram Bot API.

Если точный контракт неизвестен в рамках текущего задания:

- не выдумывай сложную реализацию;
- реализуй минимальный безопасный вариант;
- оставь понятный TODO только там, где это действительно необходимо;
- в итоговом сообщении явно укажи, что требует сверки с официальной документацией.

Не реализуй сразу весь Bot API, если текущий промпт просит только архитектуру, каркас или ограниченный набор методов.

## 12. JSON и совместимость

Для Telegram API важно аккуратное JSON-поведение.

Правила:

- используй корректные `json` tags;
- не меняй имена полей Telegram без необходимости;
- optional-поля должны корректно работать с `omitempty`, указателями или специальными типами;
- не теряй неизвестные поля, если в конкретном месте важна forward compatibility;
- не используй `map[string]any` там, где можно описать тип.

## 13. HTTP-клиент

HTTP-слой должен быть тестируемым.

Требования:

- должна быть возможность передать свой `http.Client`;
- base URL должен быть настраиваемым для тестов;
- token не должен попадать в логи;
- ошибки сети и ошибки Telegram API должны различаться;
- multipart upload должен быть реализован отдельно от обычного JSON/form запроса.

## 14. Middleware

Middleware — это цепочка вокруг обработки update.

Middleware не должен быть обязательным для простого использования.

Хороший базовый контракт:

```go
type Handler func(context.Context, *Update) error

type Middleware func(Handler) Handler
```

Но точную форму выбирай исходя из текущей архитектуры и задания.

Middleware может использоваться для:

- logging;
- recovery;
- metrics;
- auth/filtering;
- rate limiting;
- timeout;
- dependency injection через context.

## 19. Сетевое окружение пользователя

Локальная машина пользователя работает через TUN xray. Это может менять маршрутизацию локальных `curl`, `go run`, Telegram API checks, SSH, discovery local Bot API server, webhook URL и smoke scripts.

При network debug всегда явно различай:

- локальную машину пользователя;
- сервер `vk1`;
- local Telegram Bot API server;
- официальный `api.telegram.org`;
- webhook endpoint;
- SSH tunnel.

Правила диагностики:

- При сетевой ошибке сначала выясни, где выполнялась команда: локально или через `ssh vk1`.
- Если проверяется серверное окружение, предпочтительно выполнять диагностику через `ssh vk1`, потому что результат локальной проверки может отличаться из-за TUN xray.
- `localhost` и `127.0.0.1` всегда относятся к той машине, где выполняется команда. `127.0.0.1` локально не равен `127.0.0.1` на `vk1`.
- Если discovery нашёл `AIGRAM_BASE_URL=http://127.0.0.1:8081` на `vk1`, локальные smoke scripts могут требовать SSH tunnel, например проброс локального `127.0.0.1:8081` на `vk1:127.0.0.1:8081`.
- Webhook URL должен быть доступен именно тому компоненту, который отправляет webhook-запросы: official Telegram, local Telegram Bot API server или локальный smoke probe.
- Нельзя сразу менять код библиотеки при сетевой ошибке. Сначала проверь маршрутизацию, SSH tunnel, firewall/listening ports, systemd logs, `getWebhookInfo`, `DeleteWebhook`/webhook-vs-long-polling mode.
- Правило безопасности: не печатай bot token, webhook secret, token-bearing URLs и полные Bot API endpoints вида `/bot<TOKEN>/...`.

## 20. Производительность

Не оптимизируй преждевременно.

Но избегай очевидно плохих решений:

- лишнего копирования больших файлов;
